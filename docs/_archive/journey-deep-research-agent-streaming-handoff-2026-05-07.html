Grounded research: actual docs/playbooks we should copy from in principle

These are not invented patterns. They come from the platform docs and local source references inspected for this handoff.

ReferenceWhat it saysWhat AI-GOS should copy/adapt
Vercel AI SDK v6 — Stream Protocols
ai-sdk.dev/docs/ai-sdk-ui/stream-protocol		 AI SDK UI supports useChat; text streams append chunks into a response; UI message streams use SSE parts including text-start, text-delta, text-end, source-url, data-*, tool input/output, step start/finish, and finish. Backend can return result.toUIMessageStreamResponse(). Stop treating worker output as only final results. Convert research events into UI-message-compatible stream parts: visible assistant text deltas for draft writing, source parts for citations, and custom data-journey-agent/data-journey-section parts for UI state.
Vercel AI SDK v6 — Agents overview
ai-sdk.dev/docs/agents/overview		 Agents are LLMs using tools in a loop. ToolLoopAgent handles loop, context management, stopping conditions, and reusable tool configuration. Docs also say use structured workflows when repeatable outcomes need explicit control flow. Use a visible tool loop for the chat-facing agent, but do not make reveal order nondeterministic. Use structured workflow gates for the sequence: Deep Research → Market → ICP → Competitors → Offer → Synthesis. The agent can reason, but the UI reveal contract is deterministic.
Cursor Agent overview
cursor.com/docs/agent/overview		 Cursor describes Agent as instructions + tools + model. Tools include codebase/web search, file edits, terminal, browser. Cursor also has checkpoints, queued messages, and immediate messages to redirect active work. Make the user feel in control while the agent runs: visible timeline, editable artifact, interrupt/redirect path, and safe rollback/refresh-resume state. Do not hide work behind “processing.”
Cursor Plan Mode
cursor.com/docs/agent/plan-mode		 Plan Mode researches the codebase, asks clarifying questions, creates a reviewable plan, and lets the user edit the plan before building. For larger changes, precise scoping matters more than follow-up patching. For this slice, Codex should receive a locked plan and acceptance tests before implementation. No broad “make it better” prompt. Each worker gets one responsibility and explicit non-goals.
Cursor Subagents
cursor.com/docs/subagents		 Subagents are isolated contexts for long research, parallel work, specialized prompts, and independent verification. Built-in examples include Explore, Bash, Browser. Best practices: focused subagents, good descriptions, version controlled agents, independent verifier. Use five focused Codex/subagent tracks, not one mega prompt. Keep main session as orchestrator. Require each subagent to return files changed, commands run, exit codes, and PASS/FAIL evidence.
Claude Artifacts
Claude Help Center		 Artifacts are significant, self-contained content likely to be edited, iterated on, reused, and shown in a dedicated window beside chat. Users can ask Claude to modify the artifact; changes appear directly in the artifact window; multiple versions are supported. The GTM report is an artifact, not a hidden card dump. It should be born in the chat stream, render as a growing report block, then become a persistent editable artifact with chat-driven edits and versions.
AionUi local README
/Users/ammar/hermes-workspace/AionUi/readme.md		 AionUi says users “see everything the agent does,” supports multiple AI agents in one interface, parallel sessions with independent context, and Team Mode where a leader delegates to teammates and aggregates results. Adopt the Cowork feeling: visible agent work and clear user control. Parallelism can exist backstage, but the foreground narrative should be one coherent leader/agent stream.
AI coding agents playbook
~/.hermes/skills/software-development/ai-coding-agents-playbook/SKILL.md		 Human owns architecture and measurement; draw before prompting; write locked intent docs; use TDD; use parallel agents only for independent work; keep review gates; verify with tests/browser QA. Create a checked-in handoff and tests before Codex writes. Use RED/GREEN acceptance gates and a skeptical review agent. This is exactly the kind of feature that fails if assigned as a vague UI polish task.

Current AI-GOS gap map

1

Input is still URL-first

src/app/journey/page.tsx normalizes arbitrary input as a URL, so research airtable.com is not treated as a natural chat command. journey-agent-chat.tsx uses URL-specific labels/placeholders and direct submit behavior.

2

All specialists show too early

SPECIALIST_ORDER and the current assistant card render the full specialist list upfront. The desired UX is Deep Research Agent first, then one section at a time.

3

Status rows replace streamed writing

Current UI polls snapshots/activity and shows short updates. It does not render an assistant-written artifact growing in the central chat.

4

Worker deltas are ignored

The worker observes Anthropic streaming text events, but text deltas are not persisted as draft chunks. Final results are written later, which kills the ChatGPT/Claude-style experience.

5

Implementation detail leaks

The UI exposes profile extraction counts like “23 profile fields extracted.” Users need to see research evidence and report sections, not internal field counts.
