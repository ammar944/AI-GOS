The proof loop is real and useful as a dev bench, but it currently over-states production fidelity for exactly the two pillars the handoff foregrounds: the ledger and the deck gate. Closing this is a decision, not a discovery — see next moves.

What stands between here and a trustworthy, shippable deck — ordered by what to clear first.

The fork · decision

The fact ledger never runs in production

Production dispatch injects no factStore, so the ledger write no-ops, the read returns [], and the model prompt is corpus-only. The honesty/ledger machinery the harness proves is dormant on the shipped path — and the handoff's "wired" claim is the same stale-handoff trap it was created to fix.

Decide: either (a) inject createInMemoryResearchFactStore() (or a Supabase-backed store) at the dispatch seam — lab-section-dispatch.ts / lab-section-job.ts — and thread it into both prepareSectionContext and runSection deps; or (b) explicitly relabel the harness a dev bench and stop citing its ledger/gate verdicts as production-representative. (a) is the real "wire consumption to prod" move.
High

Segment miner is correct but inert — no deterministic floor

The miner surfaces clean, pre-verified segment leads, but authoring is left entirely to the model, which records them not_selected and ships empty most runs. The one deterministic authoring path (promoteDeadlineBuyerICPPersonas) explicitly skips the segment_evidence venue.

Fix (preferred #1): extend promoteDeadlineBuyerICPPersonas + add a normal-commit fallback to author segmentLabel personas directly from segment_evidence candidates (copy segmentLabel verbatim, sourceUrl from the lead). Guarantees a floor regardless of model behavior. Avoid the per-section Sonnet override — no per-section model seam exists, it's net-new infra, and it converts a deterministic-fixable problem into a probabilistic one.
