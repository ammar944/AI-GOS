## Implementation Notes

### Subcommand → Pipeline Step Mapping

| Subcommand | Calls | Description |
|------------|-------|-------------|
| `plan` | T5 (direction picker) + T6 (section planner) | Picks direction, produces plan JSON only, no HTML |
| `generate` | T7 (HTML renderer) | Takes an existing plan JSON, renders HTML |
| `full` | T5 → T6 → T7 (all three) | End-to-end: brand spec in, 3 HTML files out |
| `tweak-text` | T8 | In-place copy edits to existing HTML output |
| `tweak-regen` | T9 | Regenerates one section with a new direction seed |

### Argument Parsing

**DO NOT** use `--key=VALUE` form (e.g. `--style=VALUE`). Known gotcha: the existing `skills/landing-page/scripts/generate.ts` silently corrupts prompts when args are passed in `--key=VALUE` form. Always use space-separated form: `--key VALUE`.

Recommended approach: use `process.argv` with a minimal hand-rolled parser, or `minimist` (already likely available via devDependencies). Do NOT introduce `commander` or `yargs` — those are heavy deps not present in the reference skill pattern.

Pattern to follow — `skills/research-competitor/scripts/orchestrate.ts`:
- Reads argv
- Runs pipeline steps conditionally based on file existence (idempotent)
- Exits with appropriate code

### Exit Codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | User error (bad args, missing required flag, file not found) |
| 2 | Pipeline step failure (one of T4–T9 returned non-zero) |
| 3 | Missing environment variable (ANTHROPIC_API_KEY or OLLAMA_API_KEY not set) |

### --help Behavior

Every subcommand must support `--help` (exit 0, print usage to stdout). The top-level `cli.ts --help` should list all subcommands.

### Idempotency

For the `full` subcommand: check if plan JSON already exists in `--out` dir. If yes, skip T5+T6 and go straight to T7 (re-render). This mirrors the `orchestrate.ts` pattern of "run steps conditionally based on file existence."

### TypeScript Pattern

```typescript
// cli.ts top of file
#!/usr/bin/env npx tsx

const [,, subcommand, ...rest] = process.argv;

switch (subcommand) {
  case 'plan': await runPlan(parseArgs(rest)); break;
  case 'generate': await runGenerate(parseArgs(rest)); break;
  case 'full': await runFull(parseArgs(rest)); break;
  case 'tweak-text': await runTweakText(parseArgs(rest)); break;
  case 'tweak-regen': await runTweakRegen(parseArgs(rest)); break;
  case '--help': case '-h': printHelp(); process.exit(0); break;
  default:
    console.error(`Unknown subcommand: ${subcommand}`);
    printHelp();
    process.exit(1);
}
```

Each `runX()` function imports the relevant pipeline script from T4–T9.

## Gotchas

1. **`--key=VALUE` corrupts prompts** — always parse as `--key VALUE` (space-separated). This is a confirmed bug in the existing landing-page CLI.
2. **Two system prompts** — the existing skill has diverged prompts for CLI vs API paths. The new CLI should use a single unified prompt path from T7.
3. **Idempotency** — the `full` subcommand should not re-run LLM steps if plan JSON already exists. Check `--out` directory before calling T5/T6.
4. **`package.json` bin field** — add a `bin` entry so the CLI is invocable as `landing-page-studio` after `npm link` or install. Example: `"bin": { "landing-page-studio": "./scripts/cli.ts" }`.
5. **shebang** — first line of cli.ts must be `#!/usr/bin/env npx tsx` for direct execution.
