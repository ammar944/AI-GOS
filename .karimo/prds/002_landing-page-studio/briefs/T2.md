## Context

`BrandSpec` is the primary input contract for the landing-page-studio skill. Every other task in this skill (orchestrator, section generators, QA gates) receives a `BrandSpec` and must not need to define brand-related types themselves.

The schema must live inside `skills/landing-page-studio/assets/` because the skill is a self-contained island — no imports from `src/lib/` or outside the skill folder.

This schema is NOT passed to `generateObject()` directly as the output schema. It is an input type used to validate user-supplied brand data before the generation pipeline runs. However, apply the Anthropic-API Zod constraint defensively (see Gotchas) in case it gets wired to an LLM call later.

The BrandSpec captures:

| `logoUrl` | `string \| undefined` | Optional URL to logo asset |
| `customInstructions` | `string \| undefined` | Optional free-text override for generation |

### Schema Pattern (from existing skills)

The existing GTM skill schemas in `src/lib/gtm/skills/` use `z.object({...})` with `generateObject`. Mirror that style. Key pattern:

```typescript
export const BrandSpecSchema = z.object({
  brandName: z.string().describe("Company or product name"),
  tagline: z.string().describe("One-line positioning statement"),
  primaryColor: z.string().describe("Hex color code, e.g. #1A2B3C"),
  accentColor: z.string().describe("Hex accent color code"),
  voice: z.enum(["professional", "playful", "bold", "minimal", "warm"])
    .describe("Tone of voice for generated copy"),
  direction: z.string().describe("One of the 15 direction keys from the taxonomy"),
  industry: z.string().describe("Industry or vertical, e.g. SaaS, ecommerce"),
  targetAudience: z.string().describe("Brief description of ideal customer profile"),
  keyBenefit: z.string().describe("Primary value proposition in one sentence"),
  cta: z.string().describe("Call-to-action button text, e.g. Start free trial"),
  logoUrl: z.string().optional().describe("Optional URL to logo asset"),
  customInstructions: z.string().optional()
    .describe("Optional free-text generation override"),
});

export type BrandSpec = z.infer<typeof BrandSpecSchema>;
```

## Gotchas

**No `.min()` / `.max()` on any field — ever.**
The learned pattern from `.claude/rules/learned-patterns.md`:
> "When Anthropic rejects a schema with `properties maximum, minimum are not supported`, remove `.min()` / `.max()` from Zod number fields in schemas passed to `generateObject()`. Use `.describe()` to communicate the range to the model."

Apply this defensively to ALL fields, not just numbers. Do not add `.min(1)` to strings (even though it seems like good validation), `.max(500)` to descriptions, etc. Use `.describe("...")` to communicate constraints to both humans and LLMs.

**`direction` is a string, not an enum here.**
T3 defines the 15-direction taxonomy. At the schema layer, `direction` is typed as `z.string()` with a describe. The orchestrator will validate the direction value against T3's taxonomy at runtime. Keeping it `z.string()` avoids a circular dependency between T2 and T3.

**No imports from `src/`.**
This file must only import from `zod`. Not from `src/lib/`, not from `../../../src/`, not from any path outside `skills/landing-page-studio/`. The skill must be publishable standalone.

**Optional fields: use `.optional()`, not TypeScript union.**
`z.string().optional()` produces `string | undefined` in the inferred type. Do not write `z.union([z.string(), z.undefined()])` — it's equivalent but verbose and inconsistent with project style.
